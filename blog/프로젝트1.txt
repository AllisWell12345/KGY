HTML과 CSS, JavaScript에 대해서 열심히 배우고, 한달차에 드디어 첫 프로젝트를 진행하게 되었습니다. 첫 프로젝트이다 보니 기획 단계부터 낯설고 막막한 순간들이 많았습니다. 하지만 다행히 경험이 풍부한 팀원과 강사님의 피드백을 바탕으로 방향성을 잡았고, 대략적인 설계를 마친 뒤 본격적인 코딩에 들어갈 수 있었습니다.

 

개발을 시작하면서 제가 맡은 기능들은 UI 공통 프레임워크인 사이드바와 공통 모달창 개발, 그리고 교수 및 학생용 강의 관리 시스템과 로컬 스토리지를 활용한 상태 관리였습니다. 하지만 실제 개발을 진행하면서 예상치 못한 문제들이 발생했습니다.

 1. 문제 상황 (겪은 문제들과 발생 원인)
모달과 사이드바 등 UI 구현을 마치고 본격적으로 js를 작성하기 시작하면서 여러 페이지에서 데이터를 연동하고 동적으로 화면을 그리는 구조로 설계했습니다. 하지만 실제 개발이 진행되면서 6가지의 크고 작은 문제가 발생했습니다.


동적 UI 생성 시점 불일치: 프로필 영역만 로드되면 바로 인터벌을 종료해버려 하단의 #lecture-list 태그에 데이터를 넣지 못함
DOM 선택자 네이밍 불일치: HTML에는 id="lecture-list"인데 JS에서는 클래스인 .lecture-list로 찾아 데이터를 출력하지 못함
유동적 폰트 크기(clamp) 미적용: 화면 크기에 따라 반응하지 않고 최소값에 묶여 있거나 동적 자식 태그들이 스타일을 상속받지 못함
페이지 간 인덱스(Index) 연동 시 데이터 어긋남: 단순히 배열 순서(0, 1, 2...)만 넘겨주었다가 페이지별 데이터 정렬 로직 때문에 엉뚱한 공지사항이 열림
이벤트 버블링으로 인한 데이터 누락: <li> 내부의 <span>을 클릭했을 때 타겟이 <span>을 가리켜 부모의 data- 속성을 읽지 못함
로컬 스토리지 데이터 생성 순서 꼬임: 목업 데이터 생성 코드가 한 곳에만 있어 다른 페이지를 먼저 켜면 "등록된 강의가 없습니다"라고 뜸
이 문제는 단순 기능 오류가 아니라, 데이터의 흐름과 브라우저의 렌더링 순서를 깊게 고려하지 못해 유지보수 비용이 증가하고 데이터의 무결성이 깨지는 문제로 이어졌습니다.

 

오늘은 이 중 " 페이지 간 인덱스(Index) 연동 시 데이터 어긋남" 문제를 어떻게 해결했는지에 대해 써볼 예정입니다.

 

1-1. 문제 상황 세부: 페이지 간 인덱스(Index) 연동 시 데이터 어긋남
 

처음에는 문제들에 대해 이렇게 생각했습니다.

"코드가 잘못되었나? 데이터를 다시 지우고 만들어야 하나?"

 

하지만 문제의 본질은 단순 오타나 코드의 부족이 아니었습니다

 

"어떻게 에러를 안 나게 할까?" 가 아닌
"왜 브라우저 환경에서 동적 데이터와 UI가 어긋날까?" 로
접근을 했어야 했습니다.

 

결국 브라우저의 렌더링 시점을 확정하지 못했거나, 데이터 고유의 식별자가 아닌 가변적인 '화면의 순서(Index)'에 의존했기 때문에 발생한 문제이기 때문입니다. 목표는 단순히 에러를 없애는 것이 아니라, 어떤 상황에서도 데이터가 꼬이지 않고 안전하게 화면에 그려지는 구조를 만드는 것이었습니다.

따라서, 외부 환경이나 정렬 기준이 바뀌더라도 데이터를 안전하게 식별할 수 있는 구조가 필요했습니다. 이를 위해 인터벌 조건문을 강화하고, 배열의 순서 대신 고유한 **강의명(title)**을 키값으로 활용하여 데이터의 무결성을 해결했습니다.

 

2. 기존 구조 (단순 Index 전달 방식)

[마이페이지에서 강의 클릭]
   ↓
[해당 카드의 Index(0, 1, 2) 전달]
   ↓
[공지사항 페이지로 이동]
   ↓
[전달받은 Index로 데이터 조회]
 

기능이 “배열 순서”에 고정됨
정렬 기준이 바뀌면 엉뚱한 데이터 노출
확장성 및 안전성 없음
해당 문제를 해결하기 위해선 화면에 보이는 순서가 아니라 데이터 자체를 제어하는 구조로 변경해야 했고, 결국 인덱스 대신 고유한 강의명을 data-title 속성으로 태그에 심어 로컬 스토리지에 저장하고 이를 추적하는 방식으로 진행했습니다.

 

3. 해결 구조 (Unique Key 활용 방식)
[마이페이지에서 강의 클릭]
   ↓
[태그의 data-title(강의명) 추출]
   ↓
[LocalStorage에 강의명 저장]
   ↓
[공지사항 페이지로 이동]
   ↓
[저장된 강의명으로 일치하는 데이터 검색]
 

 

해당 구조로 데이터의 순서가 뒤바뀌더라도 고유한 강의명을 기준으로 데이터를 정확하게 찾아와 렌더링할 수 있도록 설계해보기로 했습니다.

 

4. 해결 결과
 

>> LocalStorage 데이터 정합성 유지

// 인덱스가 아닌 타이틀을 비교하여 정확한 과목 매칭
const selectedTitle = localStorage.getItem("selectedStuLecTitle");
const matchedLecture = allLectures.find(lecture => lecture.title === selectedTitle);
 

이제 각 페이지 내에서 위와 같이 안전한 방식으로 데이터를 조회하고 함수를 호출합니다.

 

예를 들어, 마이페이지에서 자바스크립트 과목을 클릭하면 순서와 상관없이 "자바스크립트"라는 고유 타이틀이 로컬 스토리지에 저장됩니다. 공지사항 페이지로 이동했을 때 이 타이틀을 기반으로 데이터를 정확히 긁어오기 때문에, 페이지가 데이터를 어떻게 다르게 정렬하든 무결성이 지켜지게 됩니다.

 

즉, 아래 흐름으로 데이터 렌더링이 진행되게 됩니다.

 

사용자 클릭
   ↓
data-title 추출 및 저장
   ↓
페이지 이동
   ↓
타이틀 기반 데이터 검색
   ↓
정확한 데이터 렌더링
 

5. 결과
5-1. 정량적 변화

데이터 누락 및 매핑 오류 발생률 0% 달성
불필요한 이벤트 리스너 중복 제거 (이벤트 위임/버블링 제어)
5-2. 정성적 변화

페이지 정렬 기준이 달라도 항상 일관된 데이터 노출
화면 간 독립적인 데이터 동기화 성공
유지보수 용이
6. 인사이트
6-1. 화면 데이터는 “순서”가 아니라 “고유 값”이다

배열의 인덱스를 믿지 말고, 변하지 않는 고유한 Key(강의명 등)로 흐름을 제어해야 한다.
6-2. UI는 항상 “브라우저 렌더링 시점”과 연결되어야 한다

태그가 그려지기도 전에 데이터를 주입하려 하면 에러가 난다. 확실한 로드 시점을 보장해야 한다.
6-3. 안전한 구조의 핵심은 “방어 코드와 명확한 이벤트 타겟”

사용자가 자식 요소를 클릭하더라도 원치 않는 버블링이 일어나지 않게 명확히 부모를 가리켜야 한다.
정리하면서...
이번 경험을 통해 개발을 할 때 단순하게 접근을 하면 안 된다는 것을 알게 되었습니다. 단순히 강의 목록을 배열 순서대로 불러와서 클릭하면 해당하는 인덱스의 페이지로 이동하게 설계하면 되겠지? 라는 안일한 생각은 당연하다는 듯, 모든 페이지에 적용할 수 있는 내용이 아니였고 심지어는 데이터가 조금만 정렬되어도 작동하지 않는다는 것을 처음 알게 되었을 땐, 꽤나 당황했습니다.


이 뿐만 아니라 계속 개발을 진행하면서도 인터벌을 대충 끝내거나, ID와 Class 선택자를 헷갈려 데이터가 안 뜨는 등 코드를 주저리 주저리 늘어놓게 되는 것은 결국 가독성 문제에서 유지보수, 성능 저하 문제로까지 이어지게 되면서 실제 개발이라는 것이 생각처럼 되지 않는다는 것을 뼈저리게 느끼게 되었습니다.


개발을 시작할 때 기획 단계는 생각보다 훨씬 중요한 단계입니다. 구조를 설계할 때 요구사항 명세에 따라 훨씬 더 세부적으로 명확하게 설계를 해야합니다. 단순한 강의 리스트를 뿌리는 작업 마저도, 데이터가 없을 때는 어떻게 대응할 것인지, 테스트 시 데이터가 먼저 안착하는 순서는 어떻게 제어할 것인지 등 생각해야 될 것이 훨씬 많다는 것을 알게 됐습니다.


이후에 첫 프로젝트가 끝나고 더 공부하던 과정에서 export와 import에 대해서 배우게 되었습니다. 이 때, 그렇다면 각 트러블슈팅을 막기 위한 방어 로직이나 유틸 함수들을 따로 정의하고 해당 함수를 export 하여 각 페이지에서 import 해온다면 좀 더 낫지 않을까? 라는 생각도 해보게 되었습니다. 이처럼 코드 리펙토링 방법은 무수히 많고, 이제는 기능 구현에만 집중하지 않고 구조를 설계하고 후에 있을 최적화까지 신경 쓸 수 있는 개발자가 되어야 겠다고 생각하게 되었습니다.
